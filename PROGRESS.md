---

## The loop — what to do every iteration

Each work session (or each `/loop` turn) follow this algorithm **exactly**:

1. **Read state.** Open this file and `ROADMAP.md`. Find the **first unchecked
   task** in roadmap order (top to bottom). That is your target. Do not skip
   ahead; do not batch unrelated tasks.
2. **Implement it fully.** Replace stubs with real logic. No `todo!()` left on
   the path you touched. Follow the conventions in `CLAUDE.md`
   (`crate::Result`, no `unwrap`/`expect` on fallible paths).
3. **Verify it (all must pass — no exceptions):**
   - `cargo fmt --check`
   - `cargo clippy --all-targets -- -D warnings`
   - `cargo build`
   - `cargo test`
   - Any acceptance check named in the task itself.
4. **If verification fails,** fix it in the same iteration. Do not check the
   box until the gate is green.
5. **Check the box** here and in `ROADMAP.md`, and add a one-line note under
   "Change log" below (what changed, what proves it works).
6. **Commit** (only if the user has authorized commits): a focused commit per
   task, message describing the slice and the proof it works.
7. **Continue** to the next unchecked task. Repeat.

**Stop only when:** all boxes checked and all
[PRODUCTION_READINESS.md](PRODUCTION_READINESS.md) gates pass. At that point,
state clearly that the project is production-ready and summarize the evidence.

**Never stop for:** a task being "big" (split it and keep going), a stub being
"good enough" (it isn't — implement it), or uncertainty about a detail (make
the smallest sound decision, note it, move on). Only pause for a genuine
blocker that needs the user (a real external dependency or a destructive/
outward-facing action requiring approval) — surface it explicitly and say what
you need.

## Definition of "production-ready" (the finish line)

The project is production-ready **only** when all of these hold — see
[PRODUCTION_READINESS.md](PRODUCTION_READINESS.md) for the full, testable list:

- A stock `mysql` CLI and at least one standard driver connect, authenticate,
  and run a realistic workload end-to-end.
- Correct handling of the wire protocol under fragmentation, large packets,
  and error paths.
- Persistence survives restart; concurrent clients are safe; transactions are
  correct.
- No panics on any client-reachable path; every error maps to a MySQL ERR
  packet. No `todo!()`/`unimplemented!()` reachable from a client.
- CI green: fmt, clippy (`-D warnings`), tests (unit + integration), and an
  end-to-end conformance suite.
- Observability (structured logs/metrics), graceful shutdown, and documented
  configuration.

- 2026-07-12 — Phase 9 (`caching_sha2_password`, MySQL 8.0's default plugin) — **completes Phase 9**. Hand-rolled SHA-256 (`auth::sha256`, FIPS 180-4), kept dependency-free like SHA-1 and verified against 4 NIST vectors including the classic 1,000,000-'a' long-input case (exercises many blocks + the 64-bit length field) — `ring` stays an implementation detail of TLS, not a general hashing API. New `auth::caching_sha2` implements the fast-auth challenge/response: the client sends `SHA256(pw) XOR SHA256(SHA256(SHA256(pw)) ++ nonce)`; the server stores only `SHA256(SHA256(pw))` and verifies without ever seeing the password by recovering the candidate `SHA256(pw)` and checking its hash with a constant-time compare. Matches the go-sql-driver reference byte-for-byte, including the concat order (stored digest **then** nonce — the reverse of native_password's `SHA1(nonce ++ …)`), which I cross-checked against a real client driver before coding. Joins execute via `hash_join`: the newly-joined table is indexed by its `ON`-column value once, then every accumulated row probes it — real algorithmic complexity, not a disguised nested-loop scan. `NULL` is never a join key on either side (`NULL = NULL` is never true in SQL), matching `WHERE`'s existing three-valued logic; `LEFT` keeps an unmatched row with the new table's columns padded `NULL`, `INNER` drops it. The Phase 5 `WHERE pk = literal` indexed fast path is fully preserved, byte-for-byte, on the no-`JOIN` path — only a `JOIN`ed query's `WHERE` is a plain linear filter (a join has no equivalent index once rows are combined, and query-planner-level predicate pushdown is out of scope for this project). `INNER`/`LEFT`/`OUTER`/`JOIN`/`ON` are now reserved keywords, matching real MySQL's own grammar — this is what lets `FROM t INNER JOIN u` parse unambiguously instead of risking `INNER` being misread as `t`'s bare alias, a real one-token-lookahead trap this parser would otherwise fall into.
