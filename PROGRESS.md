## The loop — what to do every iteration

Each work session (or each `/loop` turn) follow this algorithm **exactly**:

1. **Read state.** Open this file and `ROADMAP.md`. Find the **first unchecked
   task** in roadmap order (top to bottom). That is your target. Do not skip
   ahead; do not batch unrelated tasks.
2. **Implement it fully.** Replace stubs with real logic. No `todo!()` left on
   the path you touched. Follow the conventions in `CLAUDE.md`
   (`crate::Result`, no `unwrap`/`expect` on fallible paths).
3. **Verify it (all must pass — no exceptions):**
   - `cargo fmt --check`
   - `cargo clippy --all-targets -- -D warnings`
   - `cargo build`
   - `cargo test`
   - Any acceptance check named in the task itself.
4. **If verification fails,** fix it in the same iteration. Do not check the
   box until the gate is green.
5. **Check the box** here and in `ROADMAP.md`, and add a one-line note under
   "Change log" below (what changed, what proves it works).
6. **Commit** (only if the user has authorized commits): a focused commit per
   task, message describing the slice and the proof it works.
7. **Continue** to the next unchecked task. Repeat.

**Stop only when:** all boxes checked and all
[PRODUCTION_READINESS.md](PRODUCTION_READINESS.md) gates pass. At that point,
state clearly that the project is production-ready and summarize the evidence.

**Never stop for:** a task being "big" (split it and keep going), a stub being
"good enough" (it isn't — implement it), or uncertainty about a detail (make
the smallest sound decision, note it, move on). Only pause for a genuine
blocker that needs the user (a real external dependency or a destructive/
outward-facing action requiring approval) — surface it explicitly and say what
you need.

## Definition of "production-ready" (the finish line)

The project is production-ready **only** when all of these hold — see
[PRODUCTION_READINESS.md](PRODUCTION_READINESS.md) for the full, testable list:

- A stock `mysql` CLI and at least one standard driver connect, authenticate,
  and run a realistic workload end-to-end.
- Correct handling of the wire protocol under fragmentation, large packets,
  and error paths.
- Persistence survives restart; concurrent clients are safe; transactions are
  correct.
- No panics on any client-reachable path; every error maps to a MySQL ERR
  packet. No `todo!()`/`unimplemented!()` reachable from a client.
- CI green: fmt, clippy (`-D warnings`), tests (unit + integration), and an
  end-to-end conformance suite.
- Observability (structured logs/metrics), graceful shutdown, and documented
  configuration.

- 2026-07-12 — Phase 9 (`caching_sha2_password`, MySQL 8.0's default plugin) — **completes Phase 9**. Hand-rolled SHA-256 (`auth::sha256`, FIPS 180-4), kept dependency-free like SHA-1 and verified against 4 NIST vectors including the classic 1,000,000-'a' long-input case (exercises many blocks + the 64-bit length field) — `ring` stays an implementation detail of TLS, not a general hashing API. - 2026-07-13 — Phase 12 PD-2 (write-path architecture: dedicated log-writer thread + group commit) -- **completes PD-2**, fixing P3/P4 and amortizing D1's cost. New `storage::log_writer::LogWriter`: a plain `std::thread` owning the already-open `Log`, fed by a bounded `tokio::sync::mpsc` channel of pre-framed records (encoding stays on the calling task -- cheap and CPU-only -- so only the actual `write`/`fsync` happens on the dedicated thread); the thread blocks on `blocking_recv()` for the first queued record, then drains everything already queued via non-blocking `try_recv()` before one `write_all` + one conditional `fsync` for the whole batch. Group commit falls out of "one owner thread" for free -- deliberately not a `spawn_blocking`-per-write pool, which wouldn't coordinate at all. Callers get a `tokio::sync::oneshot` ack; a failed batch acks every waiter in it (`Error` isn't `Clone`, so each gets its own fresh `Error::Io` built from the same message). `Drop` takes the sender out of an `Option` *before* joining the thread -- joining first would deadlock, since a struct's own `Drop::drop` runs before its fields auto-drop, so the channel wouldn't actually be closed yet. Making this real required `Storage::create_table`/`insert_row`/`insert_rows` to become genuinely `async` (callers `.await` the writer's ack instead of blocking a tokio worker waiting for it): since `Storage` is used as `&dyn Storage` (`Executor` holds one) and native `async fn` in traits isn't dyn-compatible, these three return a hand-rolled `Pin<Box<dyn Future<...> + Send>>` (new `storage::BoxFuture`) -- the same mechanism `async-trait` uses, written out directly rather than adding the dependency, for exactly the three methods that touch the log; every read-only method (`scan`, `lookup_by_primary_key`, `table_schema`, `next_auto_increment`, `create_database`, ...) stayed synchronous. `create_table` used to hold `tables`' write lock across its entire log append (the one pre-PD-2 exception to "log without holding the lock"); now that the append genuinely awaits the writer thread, it follows `insert_row`'s existing check -> release -> log (no lock held) -> reacquire -> apply shape instead, since holding a `std::sync` lock across an `.await` risks stalling a worker if another task's blocking lock call lands mid-suspend. Documented (not engineered around) the one new edge case that shape introduces: two concurrent `CREATE TABLE t` calls for a brand-new name can now both pass the first check and both log a `CreateTable` record; the loser's apply sees the name taken and returns "already exists" -- harmless on replay (a redundant `CreateTable` just re-creates the same empty table), so the cost is one wasted log record on a genuinely rare race, never data loss. This cascaded into `Executor::execute`/`execute_create_table`/`execute_insert` becoming `async fn`, `Transaction::create_table`/`insert_row`/`commit` following suit (`insert_row`'s own logic stayed sync internally -- pending rows are pure in-memory buffering -- just wrapped, since the trait signature is shared), and `Connection::execute_statement`/`begin_transaction`/`commit_transaction`/`rollback_transaction` adding the resulting `.await`s -- a compile-driven refactor across ~6 files, verified call-site-by-call-site via `cargo build --all-targets` rather than trusting a manual grep alone. `~40` test call sites across `storage::engine`'s and `storage::transaction`'s unit tests plus `tests/persistence.rs`/`tests/crash.rs` converted from `#[test]` to `#[tokio::test]` with `.await` added; `query::executor`'s ~150 tests needed no per-test change at all -- they all go through one `run()` helper, which now drives the future via a throwaway current-thread runtime instead of calling the (former-sync) `execute` directly.
